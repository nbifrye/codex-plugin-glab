# GitLab イシュー開発

イシューを起点として開発を開始する：イシュー詳細の確認からブランチ作成、実装、プッシュまでを一貫して行う。

## ワークフロー

1. **���シューコンテキストの解決**
   - ユーザーがイシュー番号を提供した場合、それを直接使用する。
   - URLが提供された場合、プロジ��クトパスとIIDを抽出する。
   - `glab_issue_view <IID>` で以下を取得する：
     - タイトル、説明、ラベル、マイルストーン、担当者
     - 状態（クローズ済みの場合は警告する）

2. **イシュー内容の分析**
   - イシューの説明を読み、実装に必要な要件を整理する。
   - `glab_issue_view <IID> --comments` でコメントを確認し、追加情報や議論の結果を把握する。
   - 複雑なイシューの場合、全コメントの詳���取得には `$gitlab-issue-fetch` を使用する。
   - 実装計画をユーザーに提示し、方針を確認する。

3. **フィーチャーブランチの作成**
   - デフォルトブランチの最新を取得する：`git fetch origin <default_branch>` → `git checkout <default_branch>` → `git pull`。
   - ブランチ命名規則に従って新しいブランチを作成する：
     - 機能追加: `feature/<IID>-<slug>` （例: `feature/42-add-user-auth`）
     - バグ修正: `fix/<IID>-<slug>` （例: `fix/15-fix-login-crash`）
     - `<slug>` はイシュータイトルから生成する（小文字、スペースをハイフンに、特殊文字を除去）。
   - `git checkout -b <branch>` を実行する。

4. **開発作業の実施**
   - イシューの要件に基づきコード変更を実装する。
   - コミットメッセージにイシュー番号を含める：
     - `refs #<IID>`: イシューへの参照
     - `Closes #<IID>`: マージ時にイシューを自動クローズ
   - 変更が複数のステップに分かれる場合は、論理的な単位でコミットする。

5. **リモートへのプッシュ**
   - `git push -u origin <branch>` でリモートにプッシュする。
   - プッシュが成功したことを確認する。

6. **イシューのステータス更新（任意）**
   - 必要に応じて `glab_issue_update <IID>` でラベルやアサインを更新する。
   - 例: `In Progress` ラベルの追加、担���者の設定。
   - `glab_issue_note <IID> --message "..."` で作業開始のコメントを投稿��る（任意）。

7. **次のステップの案内**
   - MR作成が必要な場合は `$gitlab-mr-create` へのルーティングを提案する。
   - CIパイプラインの確認が必要な場合は `$gitlab-ci` を案内する。

## ブランチ命名規則

| イシュータイプ | プレフィックス | 例 |
|--------------|-------------|-----|
| 機能追加 | `feature/` | `feature/42-add-user-auth` |
| バグ修正 | `fix/` | `fix/15-fix-login-crash` |
| リファクタリング | `refactor/` | `refactor/30-cleanup-api-layer` |
| ドキュメント | `docs/` | `docs/55-update-api-docs` |
| その他 | `chore/` | `chore/60-update-dependencies` |

slug生成ルール：
- イシュータイトルを小文字に変換する。
- スペースと特殊文字をハイフンに置換する。
- 連続するハイフンを1つに圧縮する。
- 先頭・末尾のハイフンを除去する。
- 最大30文字に切り詰める。

## 出力

作成されたブランチ名、イシューの概要、実装した変更のサマリー、プッシュ状態、次のステップの提案を報告する。
